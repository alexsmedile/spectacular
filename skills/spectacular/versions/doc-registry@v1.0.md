# Doc Registry — the single source of truth for what Spectacular knows how to write

Loaded when the orchestrator needs to resolve any `spectacular <doc> <verb>` command. The registry maps each known document type to its template, slot list, mode, location, and per-doc rules.

**Adding a new doc type to Spectacular = adding an entry to this file + creating a template.** No other code changes required.

## Core principle

**One registry, many docs. Generic engine, per-doc rules.**

The engine (`grill.md` / `refine.md` / `review.md`) is doc-agnostic. It reads this registry to know:
- Which template to scaffold from
- Which slots to walk
- How to behave (`grill` interactive, `reps` per-block grill, `append` single entry, `stub` scaffold-and-exit, `freeform` agent improvises, `reference` engine-internal)
- Where the file lives
- Which per-doc rules apply (gate logic, vague-word lists, etc.)

When no rules file exists, the engine runs base behavior.

## Schema

Each entry is YAML inside a fenced block. Required fields are marked **required**.

```yaml
<doc-id>:                              # required, kebab-case, used in triggers
  template: <path>                     # required, relative to skills/spectacular/
  mode: grill | reps | append | stub | freeform | reference  # required
  location: <path>                     # required, where the file is created
  scope: project-wide | per-request | user  # required
  slots: [Slot1, Slot2, ...]           # required if mode=grill or mode=reps, omit otherwise
  snapshot-on-edit: true | false       # default false
  rules: <path>                        # optional, path to per-doc rules doc
  kit-support: true | false            # default false; whether kits can mutate this doc's slots
  description: <one-line>              # required, surfaced in `spectacular <doc> --help`
```

### Field semantics

**`template`** — markdown file the engine copies as the initial scaffold. Placeholders in the template (`<DATE>`, `<PROJECT NAME>`, slot markers) are filled by the engine using context.

**`mode`** — signals to the agent how this doc should be filled. The CLI itself never walks slots; mode tells the skill-side engine what shape of interaction is correct.

- `grill` — walk slots in order, ask one question at a time, run inline mini-refine, exit on review gate pass (PRD-style: one set of slots fills one document)
- `reps` — same grill walk, repeated per block. The agent asks "add another?" after each completed block. (ROADMAP-style: same slots filled for each version block; PERSONAS-style: same slots for each persona.) Sections can be added, edited, or skipped independently. Review gate runs per-block.
- `append` — append a single new entry to the existing file (e.g. ADR entries in DECISIONS.md); no slot loop
- `stub` — scaffold a shaped starting file and exit; no interactive walk. Template provides the structure; user fills the body. (STACK, AGENTS, PRINCIPLES, ARCHITECTURE, SPEC.)
- `freeform` — agent has full creative liberty over structure. No template imposed; no slots; agent improvises whatever shape best serves the project. Reserved for future use; no docs in v1 use this mode.
- `reference` — engine-internal doc not generated by users; loaded on demand by the engine (e.g. `migrations-contract`, `kits-contract`)

**`location`** — supports `<slug>` interpolation for per-request docs:
- `.spectacular/PRD.md` (project-wide)
- `.spectacular/requests/<slug>/PLAN.md` (per-request — slug supplied by `spectacular new`)

**`scope`** — `project-wide` files exist once at `.spectacular/` root; `per-request` files exist once per request folder; `user` files live under `$HOME` (e.g. `~/.spectacular/packs/<name>/`) and are shared across all projects on the host.

**`slots`** — ordered list of slot names. The engine walks these in order (once for `grill`, per-block for `reps`). Slot names appear as `## N. <Name>` section headings in the template.

**`snapshot-on-edit`** — when `true`, the engine snapshots the existing file (`<DOC>@vN.md`) before applying any edit. Always `true` for canonical root docs; usually `false` for per-request files.

**`rules`** — path to a markdown file with per-doc rules the engine consumes:
- Vague-word lists specific to this doc
- Custom gate checks (e.g. "Goals slot must contain number + verb + date")
- Mini-refine patterns unique to this doc type
- See `prd-rules.md` for the reference shape

**`kit-support`** — `true` only for PRD in v1. Kits (see [[kits-as-plugins]]) declare additions to this doc's slots and trigger other docs to scaffold.

```yaml
  mode: reference

  scope: skill-internal
  snapshot-on-edit: true
  description: "Schema contract for workspace-schema migration files under references/migrations/"

migration:
  template: null                       # one-off entries authored by maintainer
  mode: reference
  location: skills/spectacular/references/migrations/v<from>-to-v<to>.md
  scope: skill-internal
  snapshot-on-edit: false
  description: "Single migration spec — frontmatter (id, from, to, mechanical, reversible, apply-fn, affects) + body (detection, steps, rollback, validation). See [[migrations-contract]]."
```

> **`docs-manifest`** is project-wide (one per repo). `docs-page` is conceptually per-page but uses `per-request` scope since the engine treats `<slug>` interpolation identically. The skill's `docs new` flow supplies both `<section>` and `<slug>`.

> **No `audience` in docs-page frontmatter** — see [[docs-contract]] § Core principle. Audience is folder-level (`docs/` vs `specs/`), never per-page.

> **`scope: user`** — packs live under `$HOME` by default, not per-project. Per-project override at `<project>/.spectacular/packs/<name>/` is allowed (precedence rules: see [[packs-contract]]). This differs from the rest of the registry, which uses `project-wide` / `per-request`.

> **`rules: references/pack-rules.md`** — file does not exist in v1 (lands in request 2: convention-pack-fabricator). The registry entry is forward-declared so request 2 has a target.

## How the engine uses this

For any `spectacular <doc> <verb>` invocation:

1. Look up `<doc>` in the registry. Missing → "unknown doc type" error.
2. If `<verb>` is omitted → infer: `grill` if location is empty, `review` if filled (valid for `mode: grill` or `mode: reps`); `append` for `mode: append`; open in editor for `stub`; agent improvises for `freeform`.
3. Load `template`, `slots` (if grill/reps), `location`, `snapshot-on-edit`, `rules`.
4. Run engine verb (`grill.md` / `refine.md` / `review.md`) against this config.
5. If `rules` is set, the engine loads it and merges per-doc rules into base behavior.

## Rules file shape

A rules file is doc-specific. It contains:
- Mini-refine patterns unique to this doc (e.g. PRD's "plural-user → singular" rule)
- Custom gate checks (e.g. PLAN's "milestones before tasks" ordering)
- Vague-word lists scoped to specific slots
- Examples of good/bad answers per slot

The engine reads the rules file *if present* and applies its rules in addition to base behavior. No rules file = base behavior only.

See `prd-rules.md` as the reference example.

## Adding a new doc type

1. Pick a kebab-case ID (e.g. `verify`)
2. Create the template at `templates/<id>/base.md`
3. Add a registry entry here with required fields
4. (Optional) Create `references/<id>-rules.md` if the doc has unique rules
5. (Optional) Add the triggers `spectacular <id>` / `spectacular <id> grill` / etc. to SKILL.md routing — usually unnecessary since the generic handler catches them

No engine code changes. No new skills. No plugin manifest.

## Project-local registry overrides (v2)

Out of scope for v1, planned for v2: `.spectacular/doc-registry.yaml` in the host project would override or extend the bundled registry. Use cases: custom slot schemas, project-specific doc types, alternate templates. Until then, kit extensions (see [[kits-as-plugins]]) cover the common case for PRD customization.
