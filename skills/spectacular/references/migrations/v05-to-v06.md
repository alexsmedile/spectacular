# v0.5 → v0.6 — specs/ as always-set

## Detection

`.spectacular/specs/` does not exist as a directory (regardless of `workspace_schema:` field state).

If `specs/` already exists, this is a no-op (just bump the schema field).

## Steps

1. `mkdir -p .spectacular/specs`
2. `touch .spectacular/specs/.gitkeep` (so the empty dir tracks in git)
3. Bump `workspace_schema:` to `"0.6"` in `.spectacular/config.yaml`

## Rollback

**Not reversible** in the registry sense — no `migration_revert_v05_to_v06` function ships. The migration is mechanically *trivial* to reverse (`rm -rf .spectacular/specs && sed -i s/0.6/0.5/ .spectacular/config.yaml`), but there's no use case for downgrading a v0.6 workspace and reversibility metadata implies a callable reverse-fn — better to leave the bit at `false` than promise a function that doesn't exist.

If a downgrade is ever needed manually:

1. Verify `.spectacular/specs/` contains only `.gitkeep` (no user content); refuse if not
2. `rm -rf .spectacular/specs`
3. Set `workspace_schema:` to `"0.5"`

## Validation

- `.spectacular/specs/` exists
- `workspace_schema: "0.6"` in `config.yaml`
- Re-running migrate is a no-op

## Why this migration

v0.6.0 elevated `specs/` to always-set status (alongside `requests/`). Before that, `specs/` was optional and only scaffolded by certain kits. This migration ensures workspaces created against pre-v0.6 conventions get the folder created so doctor's always-set check passes.

This is the smallest possible migration — one directory, one frontmatter bump — and serves as a reference implementation for "trivial structural additions."
