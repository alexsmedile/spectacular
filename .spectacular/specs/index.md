# Spectacular — System Spec

## What this system is

Spectacular is an AI-native operational workspace for software projects. It ships in three layers: a `.spectacular/` directory convention, a `/spectacular` Claude Code / Codex skill that operates the workspace, and a `spectacular` CLI that bootstraps it. The skill reads frontmatter to build briefings without loading full file bodies, scaffolds requests, manages a five-state lifecycle, archives completed work, writes operational memory, exposes read-only inspection verbs for cheap cold-start, runs prototyping-stage feedback loops, and grills/refines/reviews any registered canonical doc through a single registry-driven engine.

## Capabilities

- **CLI bootstrap** — `spectacular init` scaffolds the always-set (PRD, SPEC, config, agents-file, requests/, specs/), installs the skill into `.agents/skills/spectacular/`, symlinks `.claude/skills/spectacular/`, and writes a version-pinned `skills.lock`. Idempotent — re-running fills empty stubs without overwriting content.
- **Smart-init kits** — five kits (`blank`, `coding`, `content`, `product`, `research`) declare which extra docs (`PRINCIPLES`, `ARCHITECTURE`, `ROADMAP`, `STACK`, `DECISIONS`) to scaffold via `triggers-docs`. Additive `--with <docs>` extends the kit; `--minimal` ignores kit defaults. See [[kits-contract]].
- **Doc-writing engine** *(v1.4.0+)* — one registry-driven engine (`grill`/`refine`/`review`) operates every registered canonical doc; per-doc behavior is data in `references/<doc-id>-rules.md`, agentic verbs are skill-only, mechanical verbs run in CLI. See [[specs/doc-engine]].
- **Personas (v1.3.0+)** — `PERSONAS.md` as opt-in canonical doc for proto-audience profiles + user stories. Triggered by `product` and `content` kits or `--with personas`. Lightweight (6-10 lines per persona), one file (not a folder), stories live attached to a persona. `personas-rules.md` runs grill-each per persona block. Doctor `personas` area validates structure when file exists, skips on absence.
- **Soft-DB substrates (v1.5.0+)** — `memories/`, `sessions/`, and `decisions/` are soft-folder databases: an `index.md` (catalog) plus `entries/` (one file per entry). CLI mutators: `spectacular decide "<text>"`, `spectacular remember "<text>"`, `spectacular session start|end`. Entries auto-link to the active session. Doctor adds `memory` and `sessions` areas (4-hour stale warning on `active` sessions). Coding-kit triggers updated. See [[memory-rules]], [[sessions-rules]], [[decisions-rules]].
- **`audit/` + `fixes/` soft-DB collections (v1.25.0+)** — two bug-lifecycle collections riding the same soft-DB machinery: `audit/` is the diagnosis scratchpad before a fix is planned (`spectacular audit new|list|resolve`; `resolve --into-fix` graduates an audit into a fix, copying every slot forward); `fixes/` is the verified-fix log, written only once a bug is resolved *and* verified (`spectacular fix new|list`, `--verified-by` required or the entry is marked `verified: null`). Both entry schemas carry the bug-fixing skeleton (problem → intended behavior → root cause → fix → success criteria); `fixes/` adds **Verified by** + a searchable **Signature**. `spectacular fix new --debug-job <slug>` back-links to the trace folder (v1.26.0). See [[soft-db-index]], [[audit-rules]], [[fixes-rules]].
- **Feedback-loop (v1.6.0+)** — distinct skill mode for prototyping-stage human-feedback acquisition (not benchmark, not verification — PRINCIPLES.md §9 codifies the distinction). Five-step loop: pick target → run → capture reaction → triage → resolve. `.spectacular/feedback/` substrate with `index.md` + `entries/`. CLI verbs: `feedback-loop new|list|resolve|archive`. Doctor `feedback` area. Auto-promotion of durable preference signals to memory entries — always requires explicit user confirm. Proactive surfacing only at three checkpoints (milestone tick, request → review, archive flow).
- **Ideas doctype (v1.7.0+)** — `idea` registered as first-class doc-type for parked thinking that isn't ready to become a request. CLI verbs `idea new|list|promote`. Status enum: `parked | exploring | promoted`. Promotion scaffolds a request and moves the source idea to `archive/ideas/`. Doctor `ideas` area warns at 90 days stale on `exploring` status. Mirrors `feedback` shape.
- **Read verbs (v1.8.0+)** — 11 read-only CLI verbs collapse multi-step agent workflows into single deterministic calls: list verbs (`requests`, `decisions`, `memories`, `sessions`), detail verbs (`request <slug>`, `decision <slug>`, `memory <slug>`, `sessions show <slug>`), and workspace overviews (`show <doctype>`, `summary`, `progress <slug>`, `paths`). Universal flags: `--status`, `--since <Nd|Nh|Nw>`, `--limit N`, `--all`, `--json`. Skim-by-default (frontmatter + outline + milestone progress); `--full` opt-in for raw content. Cold-start pattern: `summary → requests --active → request <slug>`.
- **Snapshot system (v1.5.0+; retention + version coupling v1.24.0+)** — snapshots live under `.spectacular/<store>/<DOC>/@v<ver>.md` (store configurable via `snapshots.folder`, default `_snapshots` since v1.24.0). The `@v<ver>` filename **couples to the doc's `version:`** so the label and the field never drift; version-less docs (e.g. `DESIGN.md`, now snapshot-able) use a plain `@v<N>` counter and aren't version-bumped. **Tiered retention** (origin `@v1` + periodic month/week + recent `keep`, default 3) bounds growth — `spectacular snapshot prune` removes the rest (git-rm if tracked, else `.trash/`), dry-run by default. Store folder, retention, and gitignore are config (`snapshots:` block). Doctor `snapshots` area validates layout, flags folder/gitignore drift + prunable accumulation; `--fix` migrates layout and the `snapshots/` → `_snapshots/` rename mechanically. See [[versioning]].
- **`spectacular status` fleet view + enforced PLAN/TASKS structure (v1.29.0+)** — bare `status` renders the active-request fleet deterministically from each `requests/<slug>/PLAN.md` frontmatter plus grep-safe body signals (the `## Goal` line, `x/total` top-level task progress with `- [~]` counted as deferred, and the current milestone = first `### M<N>` with an open top-level task), sorted active→planned; `status <slug>` prints a request card, `status --json` is the agent opt-in contract for discovering requests without opening files. This retires the hand-cached CLAUDE.md Active Requests table (generator-as-source-of-truth). It relies on an **enforced structure**: PLAN uses canonical unnumbered headings in order (`## Goal / ## Constraints / ## Milestones / ## Tasks / ## Dependencies / ## Validation / ## Deliverables`; extras allowed between); TASKS groups by `### M<N>` with flush-left `- [ ]`/`- [x]`/`- [~]` checkboxes. `doctor` (lifecycle) **errors** on structure drift for active requests (archive skipped); `doctor --fix` de-numbers legacy `## 1. Goal` headings. See [[status]], [[plan-rules]], [[tasks-rules]].
- **Lifecycle** — five states (`planned → active → review → verified → archived`) stored only in `PLAN.md` frontmatter. An optional orthogonal `hold:` field (`deferred` / `blocked`) pauses a request without adding a sixth state — `status` renders it as `planned(deferred)` (fleet), a card `(hold: …)`, and a `--json` `hold` field; `advance` refuses to move a held request until the field is cleared (supersedes the earlier `status: parked` expedient). Skill proactively detects signals (all TASKS items checked → propose `review`; stale `active` requests → flag). Archiving runs a **closure gate** (v1.28.0+): `spectacular archive` blocks on open TASKS, an unwalked VERIFY, or a missing `SPEC-DELTA.md`, each overridable once with `--override <check> --reason` (recorded in `archive_overrides:`).
- **Verification 2-of-6 rule** — `requests/<slug>/VERIFY.md` only scaffolded when 2+ of 6 complexity criteria fire; otherwise verification folds into PLAN § Validation. Canonical: [[verify-authoring]] (split from verify.md in b30; verify.md is the walk-only runtime loaded at `review → verified`); compact copy in [[plan-rules]] for scaffold/grill journeys.
- **Convention packs (v0.4.0+)** — opt-in repo-shape opinions declared via `config.yaml`'s `convention_pack:` block. Six rule categories (naming, taxonomy, root-files, gitignore, file-placement, project-types). Four scope locations (project-local → user → app-store → bundled). Three modes (suggest / scaffold / enforce). Bundled `minimal` pack + `alex-default` app-store reference. See [[packs-contract]].
- **Policy engine (v1.12.0+)** — `POLICY.md` is the practice layer paired with `PRINCIPLES.md` (theory). Policies file under nine named work-phase hooks (`@Init`, `@Planning`, `@Implementation`, `@Debugging`, `@Verification`, `@Archive`, `@Remember`, `@Snapshot`, `@SessionEnd`); on entering a phase the skill retrieves only that hook's policies and injects them — progressive disclosure applied to the rule layer. Ships always-set with 19 prefilled defaults (4 block, 15 warn); users customize/disable/add via `config.yaml`. A policy may tag the principle it enforces. Headline built-in: `understand-before-change` @ `@Implementation` gates `planned → active`. Each policy carries a one-sentence `- directive:` (the practice layer, injected verbatim at gates); hook output is tiered — warn rows print directive + principle title, block rows add the full principle line, `--full` restores paragraphs. CLI: `spectacular policy [@<hook> | --principle N] [--full]`. See [[policy-engine]].
- **Agent fleet (debug fleet v1.26.0+; build/review/verify fleet v1.29.0+)** — subagents the orchestrator delegates to, all sharing one contract: closed brief in, findings/diff/pass-fail out, **the orchestrator is the only mutator** (no agent ticks a checkbox, moves lifecycle, or writes the ledger). The fleet is a **discover / apply / review** grid across the **fix** and **build** directions, plus specialists. Fix side: **`debug-investigator`** (read-only; open bug → *where* + *why* + ranked findings, never prescribes the edit), **`debug-fixer`** (apply-only; one *closed* five-slot brief → smallest faithful diff, bounces on judgment), **`debug-researcher`** (read-only + web; known-external-bug → cited verdict). Build side: **`repo-explorer`** (read-only; map an unfamiliar subsystem before planning → seams/precedent, the build-analog of the investigator), **`spec-builder`** (apply-only; a *closed* milestone brief → diff, bounces on planning). Cross-cutting: **`code-reviewer`** (read-only; a bounded diff across five lenses → severity-ranked findings, never the fix), **`spec-reviewer`** (read-only; a doc — chiefly `specs/` — vs its rules-file rubric + currency-vs-code → punch list), **`test-verifier`** (apply-only, tests only; run a named check or write a test to a closed spec → honest pass/fail). Routed by `references/bug-workflow.md` (fix) and `references/build-workflow.md` (build). The build arc is **four phases** — **A Frame** (chain-close into a brief → route the shape: monolith-self / serialize / fan-out) · **B Decide** (inline-vs-dispatch worth-it/fan-out gate + size-the-unit) · **C Execute** (build inline / dispatch spec-builders / critical-check plan) · **D Reconcile + record** (reconcile parallel outputs → confirm + record) — with optional judgment-gated `code-reviewer` / `test-verifier` before recording. Its connective spine is the **[join] principle**: when work fans out, *how the pieces fit together is a design decision the orchestrator owns, never a builder's* — the shared contract (signature / key / protocol) is designed up front in A3, written into every brief in C2, and verified held in D1. Splitting work without specifying the join delegates a design problem, not the work. The **size-and-decompose gate** (build-workflow B2; `decompose-large-milestone` @Implementation policy) breaks a multi-phase milestone into sequential nested-`- [ ]` sub-step checkpoints built/dispatched one at a time — so a fat milestone never runs as one opaque block (visibility from sub-step boundaries, not a live trace on a running agent). The **critical-check plan** (build-workflow C3, dispatched work only) has the orchestrator name — at dispatch, while the Builder runs — the 1–2 sharpest things it will verify first on return (the risky seam / shared interface / security line); delegated work is confirmed harder than self-built (inline, the orchestrator watched the code happen; delegated, it holds only the diff). The **route-the-work step** (build-workflow A3) is the orchestrator's core maneuver *before* the mechanics fire — two axes (decomposable? one hard monolith → self; separable pieces → split; independent? coupled → serialize inline; disjoint → decompose-and-delegate) gated on benefit-of-delegation / plan-readiness / module-clarity, with monolithic-self as the default (delegation must earn it). The **reconcile step** (build-workflow D1, parallel disjoint fan-out only) confirms independently-built pieces compose — shared contract holds, seams fit, the whole integrates — before any checkbox ticks; a reconciliation mismatch is the orchestrator's to stitch (it lives in the seam between briefs, which was no single builder's job), not a builder bounce. Agent defs live in `agents/` (plugin-root source of truth; `.claude/agents/*.md` are relative symlinks). The debug arc's trace schema: one folder per job under `.spectacular/debugs/<job-slug>/`, one JSON artifact per agent turn (`job.json` spine + `investigation.json` / `research/` / `fixes/` leaves + `outcome.json`); `debugs/` is the raw pipeline (kept, never pruned), `audit/` + `fixes/` are the two distilled summaries earned at resolution. Two no-fix disposition forks (`folded-into-request`, `wont-fix`) log no `F<N>`. Backed by Principle 11 and the `@Debugging` policy set (`check-prior-fixes`, `ceremony-matches-uncertainty`, `fix-root-not-symptom`, `log-only-verified-reusable`, `use-audit-fix-verbs`). See [[bug-workflow]], [[build-workflow]], [[debug-trace]].
- **`doctor debug` (v1.26.0+)** — validates hand-written `debugs/<slug>` trace spines against the schema enums (`job.json` `status`, `outcome.json` `disposition`) and the invariant that a `wont-fix`/`folded-into-request` job logs no `F<N>`. Guards the closed-enum drift an LLM orchestrator makes hand-writing these JSON files (no `debug` write-verb exists — scaffolding + status transitions stay skill-side; only the enum/invariant validation earned a CLI verb). See [[debug-trace]].
- **Substrate doctor** — `spectacular doctor` runs read-only self-check across areas: `skill`, `workspace`, `frontmatter`, `snapshots`, `links`, `lifecycle`, `kits`, `conventions`, `specs`, `docs`, `personas`, `memory`, `sessions`, `feedback`, `ideas`, `debug`, `policies`, `vision`, `decisions`, `roadmap`. Text reports repeat all non-pass findings in a `── findings ──` block after the summary count line. `--fix` applies mechanical repairs (gitignore append, missing dirs, dangling symlinks, missing always-set stubs, pack-driven gitignore drift, legacy `current/` → `specs/` migration, snapshot layout migration + `snapshots/` → `_snapshots/` folder rename + snapshot gitignore sync, docs-page frontmatter stub injection). The `specs` area validates `SPEC-DELTA.md` integrity (MODIFIED/REMOVED quote an existing bullet; ADDED isn't a duplicate) as the primary drift signal (v1.28.0+), plus a **frontmatter schema check** on each flat `specs/<cap>.md` (b11, v1.29.0+): required keys `status, updated, summary, related`, ISO `updated`, closed status enum `draft|published|deprecated`, and `version:` required iff `status: published` — mechanical, warning-class, `index.md` skipped, `related:` resolution left to the `links` area. SPEC.md date-drift vs the newest archive (v1.18.0+) is kept as a backstop. See [[doctor]].
- **Workspace migrations (v0.6.1+, registry pattern in v0.6.2+)** — `spectacular migrate` applies pending schema migrations to bring `.spectacular/` up to date (`--dry-run` / `--to <ver>` / `--from <ver>` / `--list` flags). Migrations live as registry .md files under `skills/spectacular/references/migrations/v<from>-to-v<to>.md` with frontmatter contract (`id`, `from`, `to`, `mechanical`, `reversible`, `apply-fn`, `affects`). CLI loader resolves `apply-fn` to bash functions. `workspace_schema:` field in `config.yaml` tracks structural version. `spectacular status --against-latest` gives one-line verdict. Judgment migrations route to `/spectacular migrate` skill walk (snapshot-before-edit + y/n/q per step). Doctor's `kits` area validates chain integrity. See [[migrations-contract]] + [[migrate]].
- **CLI mutator verbs (v0.7.0+)** — verbs replace skill-side manual file edits for lifecycle mutations: `spectacular new <slug>` (scaffolds request; `--target-version` flag in v0.7.1+), `spectacular advance <slug> [--to <state>]` (advances lifecycle; scaffolds SESSION.md on `planned → active` when absent; renamed from `promote` in v1.19.0, which remains a deprecated alias), `spectacular undo [--dry-run]` (reverses the last advance/archive/idea-promote via a gitignored `.last-mutation` breadcrumb — single-level, refuses on a stale breadcrumb; v1.22.0+), `spectacular next` (read-only — names the single highest-priority next action; v1.19.0+), `spectacular snapshot <file>` (versioned copy + bump), `spectacular archive <slug>` (move + inbound link rewrite), `spectacular touch <file>` (updated: bump). Backed by shared frontmatter helpers (`fm_get`, `fm_set`, `fm_unset`, `fm_touch`). **Mutation principle:** skill orchestrates; CLI mutates. Reference docs (new-request, archive, lifecycle, versioning) updated to instruct verb usage. `remember` remains a pure skill stub (AI judgment required); `status` gained CLI-mechanical surfaces (bare fleet table / `<slug>` card / `--json`) in v1.29.0 while the richer conversational briefing stays skill-side.
- **Structured ROADMAP (v0.7.1+; refined in v0.7.2)** — `ROADMAP.md` has two layers: a **build-id ledger** (`b1..bN` → `target-version`, the single source of truth for build→version; release-level `planned/active/shipped` status distinct from request lifecycle; `tbd` = slotted-but-not-pinned sentinel) and **per-version prose blocks** carrying a precision tier (`full | themed | vision`), a required outcome slot, a 9-phase chain, an Icebox, and a tier-aware 18-check review gate. See [[specs/roadmap]] (both layers) + [[ARCHITECTURE]] (canonical ledger schema) + [[roadmap-rules]] (grill/review behavior).
- **Flat contract docs in specs/ (v0.6.1+)** — `specs/` accepts both per-capability subfolders (`specs/<capability>/SPEC.md`) AND top-level contract docs (`specs/SCHEMA-*.md`, `specs/AXIS-*.md`). Doctor validates each shape per its kind. Pattern for projects whose primary truth is on-disk contracts rather than user-facing capabilities.
- **System spec index (v0.5.0+)** — `.spectacular/SPEC.md` is the always-on, present-tense index of what's built; optional per-capability `specs/<capability>/SPEC.md` files break out detail only when a bullet outgrows one line. Replaced legacy `current/` folder (auto-migrated via `doctor specs --fix`).
- **Public docs surface — deprecated in v1.2.0, removed in v2.0.0** — Public-facing documentation work moved to the standalone [pageworks](https://github.com/alexsmedile/pageworks) skill. Spectacular keeps `doctor docs` as **discovery-only** (folder + manifest presence; install hint if pageworks missing) and prompts to hand off at `spectacular archive` for SPEC-touching requests. The `docs init|export|new|review|status` verbs are kept but emit a deprecation banner pointing to pageworks; they will be removed in v2.0.0. See [[pageworks-handoff]].
- **Two-layer task tracking** — harness `TaskCreate`/`TaskUpdate` for session-level micro-tracking (drives CLI live UI); on-disk `requests/<slug>/TASKS.md` for persistent milestone blocks. Anti-pattern: 1:1 duplication.
- **Memory** — `spectacular remember this` writes operational lessons to `.spectacular/memories/`. Git-committed, team-visible. Never to `.claude/` personal memory.
- **Versioning** — canonical docs (root layer + `SPEC.md` + `specs/<capability>/SPEC.md` + `config.yaml`) never overwritten in place. Snapshot as `<FILE>@vN.md` before edit; unversioned filename always points to current.
- **Distribution** — published as both Claude Code plugin marketplace (`alexsmedile/spectacular`) and Codex plugin marketplace. CLI installed via curl-one-liner (`cli/install.sh`).

## How to extend this file

- Add a bullet when a new capability ships (request → verified)
- Promote a bullet to `specs/<capability>/SPEC.md` when it grows past one line
- Snapshot before major rewrites: `spectacular snapshot .spectacular/SPEC.md`
