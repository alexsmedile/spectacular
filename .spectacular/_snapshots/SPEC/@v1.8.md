# Spectacular â€” System Spec

## What this system is

Spectacular is an AI-native operational workspace for software projects. It ships in three layers: a `.spectacular/` directory convention, a `/spectacular` Claude Code / Codex skill that operates the workspace, and a `spectacular` CLI that bootstraps it. The skill reads frontmatter to build briefings without loading full file bodies, scaffolds requests, manages a five-state lifecycle, archives completed work, writes operational memory, exposes read-only inspection verbs for cheap cold-start, runs prototyping-stage feedback loops, and grills/refines/reviews any registered canonical doc through a single registry-driven engine.

## Capabilities

- **CLI bootstrap** â€” `spectacular init` scaffolds the always-set (PRD, SPEC, config, agents-file, requests/, specs/), installs the skill into `.agents/skills/spectacular/`, symlinks `.claude/skills/spectacular/`, and writes a version-pinned `skills.lock`. Idempotent â€” re-running fills empty stubs without overwriting content.
- **Smart-init kits** â€” five kits (`blank`, `coding`, `content`, `product`, `research`) declare which extra docs (`PRINCIPLES`, `ARCHITECTURE`, `ROADMAP`, `STACK`, `DECISIONS`) to scaffold via `triggers-docs`. Additive `--with <docs>` extends the kit; `--minimal` ignores kit defaults. See [[kits-contract]].
- **Doc-writing engine** *(v1.4.0+)* â€” one registry-driven engine (`grill`/`refine`/`review`) operates every registered canonical doc; per-doc behavior is data in `references/<doc-id>-rules.md`, agentic verbs are skill-only, mechanical verbs run in CLI. See [[specs/doc-engine/SPEC]].
- **Personas (v1.3.0+)** â€” `PERSONAS.md` as opt-in canonical doc for proto-audience profiles + user stories. Triggered by `product` and `content` kits or `--with personas`. Lightweight (6-10 lines per persona), one file (not a folder), stories live attached to a persona. `personas-rules.md` runs grill-each per persona block. Doctor `personas` area validates structure when file exists, skips on absence.
- **Soft-DB substrates (v1.5.0+)** â€” `memory/`, `sessions/`, and `decisions/` are soft-folder databases: an `index.md` (catalog) plus `entries/` (one file per entry). CLI mutators: `spectacular decide "<text>"`, `spectacular remember "<text>"`, `spectacular session start|end`. Entries auto-link to the active session. Doctor adds `memory` and `sessions` areas (4-hour stale warning on `active` sessions). Coding-kit triggers updated. See [[memory-rules]], [[sessions-rules]], [[decisions-rules]].
- **`audit/` + `fixes/` soft-DB collections (v1.25.0+)** â€” two bug-lifecycle collections riding the same soft-DB machinery: `audit/` is the diagnosis scratchpad before a fix is planned (`spectacular audit new|list|resolve`; `resolve --into-fix` graduates an audit into a fix, copying every slot forward); `fixes/` is the verified-fix log, written only once a bug is resolved *and* verified (`spectacular fix new|list`, `--verified-by` required or the entry is marked `verified: null`). Both entry schemas carry the bug-fixing skeleton (problem â†’ intended behavior â†’ root cause â†’ fix â†’ success criteria); `fixes/` adds **Verified by** + a searchable **Signature**. `spectacular fix new --debug-job <slug>` back-links to the trace folder (v1.26.0). See [[soft-db-index]], [[audit-rules]], [[fixes-rules]].
- **Feedback-loop (v1.6.0+)** â€” distinct skill mode for prototyping-stage human-feedback acquisition (not benchmark, not verification â€” PRINCIPLES.md Â§9 codifies the distinction). Five-step loop: pick target â†’ run â†’ capture reaction â†’ triage â†’ resolve. `.spectacular/feedback/` substrate with `index.md` + `entries/`. CLI verbs: `feedback-loop new|list|resolve|archive`. Doctor `feedback` area. Auto-promotion of durable preference signals to memory entries â€” always requires explicit user confirm. Proactive surfacing only at three checkpoints (milestone tick, request â†’ review, archive flow).
- **Ideas doctype (v1.7.0+)** â€” `idea` registered as first-class doc-type for parked thinking that isn't ready to become a request. CLI verbs `idea new|list|promote`. Status enum: `parked | exploring | promoted`. Promotion scaffolds a request and moves the source idea to `archive/ideas/`. Doctor `ideas` area warns at 90 days stale on `exploring` status. Mirrors `feedback` shape.
- **Read verbs (v1.8.0+)** â€” 11 read-only CLI verbs collapse multi-step agent workflows into single deterministic calls: list verbs (`requests`, `decisions`, `memories`, `sessions`), detail verbs (`request <slug>`, `decision <slug>`, `memory <slug>`, `sessions show <slug>`), and workspace overviews (`show <doctype>`, `summary`, `progress <slug>`, `paths`). Universal flags: `--status`, `--since <Nd|Nh|Nw>`, `--limit N`, `--all`, `--json`. Skim-by-default (frontmatter + outline + milestone progress); `--full` opt-in for raw content. Cold-start pattern: `summary â†’ requests --active â†’ request <slug>`.
- **Snapshot system (v1.5.0+; retention + version coupling v1.24.0+)** â€” snapshots live under `.spectacular/<store>/<DOC>/@v<ver>.md` (store configurable via `snapshots.folder`, default `_snapshots` since v1.24.0). The `@v<ver>` filename **couples to the doc's `version:`** so the label and the field never drift; version-less docs (e.g. `DESIGN.md`, now snapshot-able) use a plain `@v<N>` counter and aren't version-bumped. **Tiered retention** (origin `@v1` + periodic month/week + recent `keep`, default 3) bounds growth â€” `spectacular snapshot prune` removes the rest (git-rm if tracked, else `.trash/`), dry-run by default. Store folder, retention, and gitignore are config (`snapshots:` block). Doctor `snapshots` area validates layout, flags folder/gitignore drift + prunable accumulation; `--fix` migrates layout and the `snapshots/` â†’ `_snapshots/` rename mechanically. See [[versioning]].
- **Lifecycle** â€” five states (`planned â†’ active â†’ review â†’ verified â†’ archived`) stored only in `PLAN.md` frontmatter. Skill proactively detects signals (all TASKS items checked â†’ propose `review`; stale `active` requests â†’ flag).
- **Verification 2-of-6 rule** â€” `requests/<slug>/VERIFY.md` only scaffolded when 2+ of 6 complexity criteria fire; otherwise verification folds into PLAN Â§ Validation. See [[verify]] (Part 2; the trio merged into one reference in v1.20.0).
- **Convention packs (v0.4.0+)** â€” opt-in repo-shape opinions declared via `config.yaml`'s `convention_pack:` block. Six rule categories (naming, taxonomy, root-files, gitignore, file-placement, project-types). Four scope locations (project-local â†’ user â†’ app-store â†’ bundled). Three modes (suggest / scaffold / enforce). Bundled `minimal` pack + `alex-default` app-store reference. See [[packs-contract]].
- **Policy engine (v1.12.0+)** â€” `POLICY.md` is the practice layer paired with `PRINCIPLES.md` (theory). Policies file under nine named work-phase hooks (`@Init`, `@Planning`, `@Implementation`, `@Debugging`, `@Verification`, `@Archive`, `@Remember`, `@Snapshot`, `@SessionEnd`); on entering a phase the skill retrieves only that hook's policies and injects them â€” progressive disclosure applied to the rule layer. Ships always-set with 19 prefilled defaults (4 block, 15 warn); users customize/disable/add via `config.yaml`. A policy may tag the principle it enforces. Headline built-in: `understand-before-change` @ `@Implementation` gates `planned â†’ active`. CLI: `spectacular policy [@<hook> | --principle N]`. See [[policy-engine]].
- **Debug agent fleet (v1.26.0+)** â€” three subagents the orchestrator delegates to during a bug flow: **`debug-investigator`** (read-only; discovers *where* + *why* on an open bug, returns ranked findings + a plausible-solution space, never prescribes the literal edit), **`debug-fixer`** (apply-only; executes one *closed* five-slot brief â€” smallest faithful diff, local style, an operation-care gradient add<editâ‰ˆpatch<delete, risk-scaled verify â€” and bounces the moment execution turns to judgment), and **`debug-researcher`** (read-only; searches forums/docs/issues for known-external bugs, returns a cited verdict). The orchestrator is the only mutator and the single writer of the trace spine + the audit/fixes ledger; each agent writes only its own trace artifact. Governed by `references/bug-workflow.md`'s Steps 0â†’3 (seen-it-before â†’ ceremony gate â†’ fan-out gate â†’ open the job â†’ investigate/plan/fix â†’ resolve + log), with two no-fix disposition forks at Step 2b â€” `folded-into-request` (findings are design work too big to close) and `wont-fix` (a real, closeable fix the orchestrator deliberately declines) â€” neither logs an `F<N>`. Trace schema: one folder per job under `.spectacular/debug/<job-slug>/`, one JSON artifact per agent turn (`job.json` spine + `investigation.json` / `research/` / `fixes/` leaves + `outcome.json`); `debug/` is the raw pipeline (kept, never pruned), `audit/` + `fixes/` are the two distilled summaries earned at resolution. Backed by Principle 11 ("earn each step") and the `@Debugging` policy set (`check-prior-fixes`, `ceremony-matches-uncertainty`, `fix-root-not-symptom`, `log-only-verified-reusable`, `use-audit-fix-verbs`). See [[bug-workflow]], [[debug-trace]].
- **`doctor debug` (v1.26.0+)** â€” validates hand-written `debug/<slug>` trace spines against the schema enums (`job.json` `status`, `outcome.json` `disposition`) and the invariant that a `wont-fix`/`folded-into-request` job logs no `F<N>`. Guards the closed-enum drift an LLM orchestrator makes hand-writing these JSON files (no `debug` write-verb exists â€” scaffolding + status transitions stay skill-side; only the enum/invariant validation earned a CLI verb). See [[debug-trace]].
- **Substrate doctor** â€” `spectacular doctor` runs read-only self-check across areas: `skill`, `workspace`, `frontmatter`, `snapshots`, `links`, `lifecycle`, `kits`, `conventions`, `specs`, `docs`, `personas`, `memory`, `sessions`, `feedback`, `ideas`, `debug`, `policies`, `vision`, `decisions`, `roadmap`. `--fix` applies mechanical repairs (gitignore append, missing dirs, dangling symlinks, missing always-set stubs, pack-driven gitignore drift, legacy `current/` â†’ `specs/` migration, snapshot layout migration + `snapshots/` â†’ `_snapshots/` folder rename + snapshot gitignore sync, docs-page frontmatter stub injection). The `specs` area also flags SPEC.md date-drift vs the newest archive (v1.18.0+) â€” a heuristic blind to work that ships outside the request lifecycle (see `b11: spec-audit-mode` on the roadmap). See [[doctor]].
- **Workspace migrations (v0.6.1+, registry pattern in v0.6.2+)** â€” `spectacular migrate` applies pending schema migrations to bring `.spectacular/` up to date (`--dry-run` / `--to <ver>` / `--from <ver>` / `--list` flags). Migrations live as registry .md files under `skills/spectacular/references/migrations/v<from>-to-v<to>.md` with frontmatter contract (`id`, `from`, `to`, `mechanical`, `reversible`, `apply-fn`, `affects`). CLI loader resolves `apply-fn` to bash functions. `workspace_schema:` field in `config.yaml` tracks structural version. `spectacular status --against-latest` gives one-line verdict. Judgment migrations route to `/spectacular migrate` skill walk (snapshot-before-edit + y/n/q per step). Doctor's `kits` area validates chain integrity. See [[migrations-contract]] + [[migrate]].
- **CLI mutator verbs (v0.7.0+)** â€” verbs replace skill-side manual file edits for lifecycle mutations: `spectacular new <slug>` (scaffolds request; `--target-version` flag in v0.7.1+), `spectacular advance <slug> [--to <state>]` (advances lifecycle; renamed from `promote` in v1.19.0, which remains a deprecated alias), `spectacular undo [--dry-run]` (reverses the last advance/archive/idea-promote via a gitignored `.last-mutation` breadcrumb â€” single-level, refuses on a stale breadcrumb; v1.22.0+), `spectacular next` (read-only â€” names the single highest-priority next action; v1.19.0+), `spectacular snapshot <file>` (versioned copy + bump), `spectacular archive <slug>` (move + inbound link rewrite), `spectacular touch <file>` (updated: bump). Backed by shared frontmatter helpers (`fm_get`, `fm_set`, `fm_unset`, `fm_touch`). **Mutation principle:** skill orchestrates; CLI mutates. Reference docs (new-request, archive, lifecycle, versioning) updated to instruct verb usage. Only `status` and `remember` remain as skill stubs (AI judgment required).
- **Structured ROADMAP (v0.7.1+; refined in v0.7.2)** â€” `ROADMAP.md` has two layers: a **build-id ledger** (`b1..bN` â†’ `target-version`, the single source of truth for buildâ†’version; release-level `planned/active/shipped` status distinct from request lifecycle; `tbd` = slotted-but-not-pinned sentinel) and **per-version prose blocks** carrying a precision tier (`full | themed | vision`), a required outcome slot, a 9-phase chain, an Icebox, and a tier-aware 18-check review gate. See [[specs/roadmap/SPEC]] (both layers) + [[ARCHITECTURE]] (canonical ledger schema) + [[roadmap-rules]] (grill/review behavior).
- **Flat contract docs in specs/ (v0.6.1+)** â€” `specs/` accepts both per-capability subfolders (`specs/<capability>/SPEC.md`) AND top-level contract docs (`specs/SCHEMA-*.md`, `specs/AXIS-*.md`). Doctor validates each shape per its kind. Pattern for projects whose primary truth is on-disk contracts rather than user-facing capabilities.
- **System spec index (v0.5.0+)** â€” `.spectacular/SPEC.md` is the always-on, present-tense index of what's built; optional per-capability `specs/<capability>/SPEC.md` files break out detail only when a bullet outgrows one line. Replaced legacy `current/` folder (auto-migrated via `doctor specs --fix`).
- **Public docs surface â€” deprecated in v1.2.0, removed in v2.0.0** â€” Public-facing documentation work moved to the standalone [pageworks](https://github.com/alexsmedile/pageworks) skill. Spectacular keeps `doctor docs` as **discovery-only** (folder + manifest presence; install hint if pageworks missing) and prompts to hand off at `spectacular archive` for SPEC-touching requests. The `docs init|export|new|review|status` verbs are kept but emit a deprecation banner pointing to pageworks; they will be removed in v2.0.0. See [[pageworks-handoff]].
- **Two-layer task tracking** â€” harness `TaskCreate`/`TaskUpdate` for session-level micro-tracking (drives CLI live UI); on-disk `requests/<slug>/TASKS.md` for persistent milestone blocks. Anti-pattern: 1:1 duplication.
- **Memory** â€” `spectacular remember this` writes operational lessons to `.spectacular/memory/`. Git-committed, team-visible. Never to `.claude/` personal memory.
- **Versioning** â€” canonical docs (root layer + `SPEC.md` + `specs/<capability>/SPEC.md` + `config.yaml`) never overwritten in place. Snapshot as `<FILE>@vN.md` before edit; unversioned filename always points to current.
- **Distribution** â€” published as both Claude Code plugin marketplace (`alexsmedile/spectacular`) and Codex plugin marketplace. CLI installed via curl-one-liner (`cli/install.sh`).
