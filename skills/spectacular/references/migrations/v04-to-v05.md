# v0.4 → v0.5 — current/ → specs/

## Detection

`.spectacular/current/` exists AND `.spectacular/specs/` does not exist.

If neither exists, this migration is a no-op (workspace was scaffolded post-v0.5 but never had `workspace_schema:` set; just bump the field).

If both exist, the migration refuses with a clear error: this is a broken workspace state requiring manual resolution. See doctor's `specs` area for the merge-required warning.

## Steps

1. `mv .spectacular/current .spectacular/specs` (preserves contents verbatim)
2. Bump `workspace_schema:` to `"0.5"` in `.spectacular/config.yaml`

## Rollback

**Not reversible.** The rename is mechanically reversible (`mv .spectacular/specs .spectacular/current`) but downstream code in v0.5+ assumes `specs/`, so reverting the directory without reverting the CLI would break doctor. Stays one-way.

## Validation

- `.spectacular/specs/` exists with contents preserved
- `.spectacular/current/` no longer exists
- `workspace_schema: "0.5"` in `config.yaml`
- Re-running migrate is a no-op

## Why this migration

In v0.5.0, the `current/` folder was renamed to `specs/` to better reflect its role: it holds present-tense system specifications, not "current work in progress." See `.spectacular/archive/spec-rename/PLAN.md` for the full rationale.

Before this migration framework (v0.6.1+), this rename was implemented as a one-off mechanical fix in doctor's `check_specs` area. The framework absorbs it into the standard chain so future structural changes follow the same pattern.
