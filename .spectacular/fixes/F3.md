# F3 — trailing-conditional exit-1 recurred in new audit/fix verbs

## Problem
`spectacular fix new` (no `--verified-by`), `audit resolve` (no `--into-fix`), and `fix list` (non-empty) all returned exit 1 despite succeeding — the same bug class as [F2].

## Intended behavior
Every `cmd_*` returns 0 on success. Newly-written verbs should not reintroduce a bug already logged and fixed.

## Root cause
Each function's last statement was a short-circuit `[[ ... ]] && info` or `$flag && info` that evaluates false on the common path, so the function returned 1. Dispatched via `exit $?`, the nonzero propagates as a false failure. Identical mechanism to F2 — the fix knowledge existed but wasn't applied to new code.

## Fix
Added explicit `return 0` to `cmd_fix_new`, `cmd_audit_resolve`, `cmd_fix_list` (`cli/spectacular`). Same remedy as F2.

## Success criteria
All three verbs exit 0 on their common (conditional-false) path.

## Verified by
`tests/cli/audit-fix.test.sh` → `scenario_verified_gate` asserts `fix new` without `--verified-by` exits 0. Green.

## Signature
Same as F2 — any bash `cmd_*` ending in `[[ cond ]] && cmd` returns 1 when `cond` is false. **Lesson:** when writing a new `cmd_*`, end it with `return 0`. This entry exists because the F2 fix wasn't generalized into a habit — exactly the self-learning gap fixes/ is meant to close.
