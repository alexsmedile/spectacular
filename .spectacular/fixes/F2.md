# F2 — flat-mode `decide` returns exit 1 despite success

## Problem
`spectacular decide "..."` printed `✓ appended decision` and *did* persist to DECISIONS.md, but returned exit code 1 — reading as "reported success but didn't persist" to callers checking `$?`.

## Intended behavior
A successful `decide` returns exit 0. The exit code must reflect whether the append succeeded, not the truthiness of an unrelated trailing conditional.

## Root cause
The last statement of `cmd_decide`'s flat branch was `[[ -n "$session" ]] && info ...`. With no open session, `$session` is empty → the `[[ ]]` test is false → the `&&` short-circuits → the compound statement returns 1 → the function returns 1. The append had already succeeded; only the exit code lied.

## Fix
`cli/spectacular` ~line 1273: add an explicit `return 0` after the `[[ -n "$session" ]] &&` line so the successful append determines the exit code, not the trailing conditional.

## Success criteria
`decide` with no open session exits 0; the decision is still persisted.

## Verified by
`tests/cli/decide.test.sh` → `scenario_flat` runs `decide` with no open session and asserts exit 0 (was 1). Green.

## Signature
A bash function whose final line is `[[ cond ]] && cmd` or `$flag && cmd` will return 1 whenever `cond`/`flag` is false — even after the real work succeeded. Any `cmd_*` dispatched via `exit $?` is affected. Fix: explicit `return 0` at function end.
