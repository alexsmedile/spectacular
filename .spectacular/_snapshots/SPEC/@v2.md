# Spectacular — System Spec

## What this system is

Spectacular is an AI-native operational workspace for software projects. It ships in three layers: a `.spectacular/` directory convention, a `/spectacular` Claude Code / Codex skill that operates the workspace, and a `spectacular` CLI that bootstraps it. The skill reads frontmatter to build briefings without loading full file bodies, scaffolds requests, manages a five-state lifecycle, archives completed work, writes operational memory, exposes read-only inspection verbs for cheap cold-start, runs prototyping-stage feedback loops, and grills/refines/reviews any registered canonical doc through a single registry-driven engine.

## Capabilities

- **CLI bootstrap** — `spectacular init` scaffolds the always-set (PRD, SPEC, config, agents-file, requests/, specs/), installs the skill into `.agents/skills/spectacular/`, symlinks `.claude/skills/spectacular/`, and writes a version-pinned `skills.lock`. Idempotent — re-running fills empty stubs without overwriting content.
- **Smart-init kits** — five kits (`blank`, `coding`, `content`, `product`, `research`) declare which extra docs (`PRINCIPLES`, `ARCHITECTURE`, `ROADMAP`, `STACK`, `DECISIONS`) to scaffold via `triggers-docs`. Additive `--with <docs>` extends the kit; `--minimal` ignores kit defaults. See [[kits-contract]].
- **Doc-writing skill** *(v1.4.0+)* — `grill.md` + `refine.md` + `review.md` skill flows handle any canonical doc verb (`spectacular <doc> grill|refine|review`). Each doc has a `references/<doc-id>-rules.md` file whose **frontmatter** declares dispatch (mode, slots, template, location, scope, snapshot-on-edit) and whose **body** declares per-doc prompts and gate checks. [[doc-index]] is a human catalog (not parsed for behavior). Modes: `grill` / `grill-wide` / `grill-each` / `grill-loop` / `append` / `stub` / `freeform` / `reference`. Registered docs (v1.7.0, 18 total): `prd`, `spec`, `plan`, `tasks`, `principles`, `architecture`, `roadmap`, `stack`, `agents`, `decisions`, `memory`, `sessions`, `personas`, `feedback`, `idea`, `convention-pack`, `docs-manifest`, `docs-page`. Agentic verbs (`grill`, `refine`) are skill-only — CLI redirects with a friendly message; mechanical verbs (`new`, `archive`, `snapshot`, `init`, `doctor`, `pack`, `migrate`) run in CLI.
- **Personas (v1.3.0+)** — `PERSONAS.md` as opt-in canonical doc for proto-audience profiles + user stories. Triggered by `product` and `content` kits or `--with personas`. Lightweight (6-10 lines per persona), one file (not a folder), stories live attached to a persona. `personas-rules.md` runs grill-each per persona block. Doctor `personas` area validates structure when file exists, skips on absence.
- **Soft-DB substrates (v1.5.0+)** — `memory/`, `sessions/`, and `decisions/` are soft-folder databases: an `index.md` (catalog) plus `entries/` (one file per entry). CLI mutators: `spectacular decide "<text>"`, `spectacular remember "<text>"`, `spectacular session start|end`. Entries auto-link to the active session. Doctor adds `memory` and `sessions` areas (4-hour stale warning on `active` sessions). Coding-kit triggers updated. See [[memory-rules]], [[sessions-rules]], [[decisions-rules]].
- **Feedback-loop (v1.6.0+)** — distinct skill mode for prototyping-stage human-feedback acquisition (not benchmark, not verification — PRINCIPLES.md §9 codifies the distinction). Five-step loop: pick target → run → capture reaction → triage → resolve. `.spectacular/feedback/` substrate with `index.md` + `entries/`. CLI verbs: `feedback-loop new|list|resolve|archive`. Doctor `feedback` area. Auto-promotion of durable preference signals to memory entries — always requires explicit user confirm. Proactive surfacing only at three checkpoints (milestone tick, request → review, archive flow).
- **Ideas doctype (v1.7.0+)** — `idea` registered as first-class doc-type for parked thinking that isn't ready to become a request. CLI verbs `idea new|list|promote`. Status enum: `parked | exploring | promoted`. Promotion scaffolds a request and moves the source idea to `archive/ideas/`. Doctor `ideas` area warns at 90 days stale on `exploring` status. Mirrors `feedback` shape.
- **Read verbs (v1.8.0+)** — 11 read-only CLI verbs collapse multi-step agent workflows into single deterministic calls: list verbs (`requests`, `decisions`, `memories`, `sessions`), detail verbs (`request <slug>`, `decision <slug>`, `memory <slug>`, `sessions show <slug>`), and workspace overviews (`show <doctype>`, `summary`, `progress <slug>`, `paths`). Universal flags: `--status`, `--since <Nd|Nh|Nw>`, `--limit N`, `--all`, `--json`. Skim-by-default (frontmatter + outline + milestone progress); `--full` opt-in for raw content. Cold-start pattern: `summary → requests --active → request <slug>`.
- **Snapshot tidy (v1.5.0+)** — root-level `<DOC>@vN.md` snapshots live under `.spectacular/snapshots/<DOC>/@v<N>.md` (not scattered at root). Doctor `snapshots` area validates the layout; `--fix` migrates pre-v1.5.0 layouts mechanically.
- **Lifecycle** — five states (`planned → active → review → verified → archived`) stored only in `PLAN.md` frontmatter. Skill proactively detects signals (all TASKS items checked → propose `review`; stale `active` requests → flag).
- **Verification 2-of-6 rule** — `requests/<slug>/VERIFY.md` only scaffolded when 2+ of 6 complexity criteria fire; otherwise verification folds into PLAN § Validation. See [[verification]].
- **Convention packs (v0.4.0+)** — opt-in repo-shape opinions declared via `config.yaml`'s `convention_pack:` block. Six rule categories (naming, taxonomy, root-files, gitignore, file-placement, project-types). Four scope locations (project-local → user → app-store → bundled). Three modes (suggest / scaffold / enforce). Bundled `minimal` pack + `alex-default` app-store reference. See [[packs-contract]].
- **Substrate doctor** — `spectacular doctor` runs read-only self-check across areas: `skill`, `workspace`, `frontmatter`, `snapshots`, `links`, `lifecycle`, `kits`, `conventions`, `specs`, `docs`, `personas`, `memory`, `sessions`, `feedback`, `ideas`. `--fix` applies mechanical repairs (gitignore append, missing dirs, dangling symlinks, missing always-set stubs, pack-driven gitignore drift, legacy `current/` → `specs/` migration, snapshot layout migration to `snapshots/<DOC>/@v<N>.md`, docs-page frontmatter stub injection). See [[doctor]].
- **Workspace migrations (v0.6.1+, registry pattern in v0.6.2+)** — `spectacular migrate` applies pending schema migrations to bring `.spectacular/` up to date (`--dry-run` / `--to <ver>` / `--from <ver>` / `--list` flags). Migrations live as registry .md files under `skills/spectacular/references/migrations/v<from>-to-v<to>.md` with frontmatter contract (`id`, `from`, `to`, `mechanical`, `reversible`, `apply-fn`, `affects`). CLI loader resolves `apply-fn` to bash functions. `workspace_schema:` field in `config.yaml` tracks structural version. `spectacular status --against-latest` gives one-line verdict. Judgment migrations route to `/spectacular migrate` skill walk (snapshot-before-edit + y/n/q per step). Doctor's `kits` area validates chain integrity. See [[migrations-contract]] + [[migrate]].
- **CLI mutator verbs (v0.7.0+)** — five verbs replace skill-side manual file edits for lifecycle mutations: `spectacular new <slug>` (scaffolds request; `--target-version` flag in v0.7.1+), `spectacular promote <slug> [--to <state>]` (advances lifecycle), `spectacular snapshot <file>` (versioned copy + bump), `spectacular archive <slug>` (move + inbound link rewrite), `spectacular touch <file>` (updated: bump). Backed by shared frontmatter helpers (`fm_get`, `fm_set`, `fm_touch`). **Mutation principle:** skill orchestrates; CLI mutates. Reference docs (new-request, archive, lifecycle, versioning) updated to instruct verb usage. Only `status` and `remember` remain as skill stubs (AI judgment required).
- **Structured ROADMAP (v0.7.1+; refined in v0.7.2)** — `ROADMAP.md` is `mode: structured` with per-version blocks. Each block has a **precision tier** (`full | themed | vision`) capturing the natural gradient (active = detailed, long-term = fuzzy). Tier-aware grill + tier-aware 18-check review gate. **9-phase chain**: `intent → discover → prototype → spec-refine → mvp → iterate → test → release-prep → release` with skip notation. **3 meta-phase aliases** (`discover | build | release`) accepted alongside specific phase values — start coarse, refine as work crystallizes. **Prototype** is broadly defined: any artifact validating a decision against real tooling or against the user (schemas, mocks, wireframes, sample CLI output). **Outcome slot** (v0.7.2+) is required for full + themed tiers — the goal between Phase and Scope-in (Pichler/Torres/Cagan/Gilad convergent: the #1 missing slot in feature-list roadmaps). **Icebox section** (renamed from Bucket list in v0.7.2 to match GitHub Projects / Pivotal / Linear idiom) for unbound ideas, promoted via 4-step ritual. Gate guards: dates forbidden in themed/vision blocks (Cagan's "#1 sin"); soft warning at 8+ full-tier blocks; scope-out push when scope-in ≥4. Doctor flags pre-v0.7.1 freeform shape AND pre-v0.7.2 "Bucket list" naming as info. See [[roadmap-overrides]].
- **Flat contract docs in specs/ (v0.6.1+)** — `specs/` accepts both per-capability subfolders (`specs/<capability>/SPEC.md`) AND top-level contract docs (`specs/SCHEMA-*.md`, `specs/AXIS-*.md`). Doctor validates each shape per its kind. Pattern for projects whose primary truth is on-disk contracts rather than user-facing capabilities.
- **System spec index (v0.5.0+)** — `.spectacular/SPEC.md` is the always-on, present-tense index of what's built; optional per-capability `specs/<capability>/SPEC.md` files break out detail only when a bullet outgrows one line. Replaced legacy `current/` folder (auto-migrated via `doctor specs --fix`).
- **Public docs surface — deprecated in v1.2.0, removed in v2.0.0** — Public-facing documentation work moved to the standalone [pageworks](https://github.com/alexsmedile/pageworks) skill. Spectacular keeps `doctor docs` as **discovery-only** (folder + manifest presence; install hint if pageworks missing) and prompts to hand off at `spectacular archive` for SPEC-touching requests. The `docs init|export|new|review|status` verbs are kept but emit a deprecation banner pointing to pageworks; they will be removed in v2.0.0. See [[pageworks-handoff]].
- **Two-layer task tracking** — harness `TaskCreate`/`TaskUpdate` for session-level micro-tracking (drives CLI live UI); on-disk `requests/<slug>/TASKS.md` for persistent milestone blocks. Anti-pattern: 1:1 duplication.
- **Memory** — `spectacular remember this` writes operational lessons to `.spectacular/memory/`. Git-committed, team-visible. Never to `.claude/` personal memory.
- **Versioning** — canonical docs (root layer + `SPEC.md` + `specs/<capability>/SPEC.md` + `config.yaml`) never overwritten in place. Snapshot as `<FILE>@vN.md` before edit; unversioned filename always points to current.
- **Distribution** — published as both Claude Code plugin marketplace (`alexsmedile/spectacular`) and Codex plugin marketplace. CLI installed via curl-one-liner (`cli/install.sh`).

## How to extend this file

- Add a bullet when a new capability ships (request → verified)
- Promote a bullet to `specs/<capability>/SPEC.md` when it grows past one line
- Snapshot before major rewrites: `spectacular snapshot .spectacular/SPEC.md`
