# Structured roadmap

> Promoted from `SPEC.md` in v1.10.0 (was a single dense bullet). The index keeps a one-line pointer; the behavioral rules live in `references/roadmap-rules.md` and the artifact shape lives here.

`ROADMAP.md` has **two layers**: a **ledger table** at the top (the compact build→version index — schema below) and **per-version prose blocks** below it (the precision-graded planning detail). The ledger is the machine-readable spine; the prose blocks are the human narrative.

## Purpose

`ROADMAP.md` is a structured planning artifact (`mode: structured`), not a freeform list. It captures *what's next* with a deliberate **precision gradient**: near-term work is detailed, long-term work is just direction. This prevents the two failure modes of feature-list roadmaps — false precision on far-out work, and missing intent on near work. Detail for in-flight work lives in `requests/<slug>/`; shipped history lives in `CHANGELOG.md`; the roadmap is the bridge between them.

## The ledger (build → version)

The ledger is a single markdown table at the **top of `ROADMAP.md`**, above the first version block. It is the **only place a target version number is written** — request frontmatter carries a stable `build:` id instead, and all prose references requests by slug or build id.

| Concept | What it is |
|---|---|
| **Build id** (`b1`, `b2`, …) | Monotonic, immutable counter stamped on a request at `spectacular new` (increments `last_build:` in `config.yaml`). A request's permanent identity in the roadmap; never changes even when its target version shifts. |
| **`target-version`** | The version a build is slotted for (`v1.10.0`). **The single source of truth** — no version number lives anywhere else (not PLAN frontmatter, not prose, not milestone text). Reslotting a request is a one-row edit here. |
| **`tbd`** | The legitimate `target-version` value for a build that is **slotted but not version-pinned yet** — work is real and prioritized, but which release it lands in isn't decided. Distinct from a `<TBD>` *placeholder* (which the review gate rejects in prose slots): `tbd` in the ledger column is a committed sentinel, not an unfilled blank. A build moves from `tbd` to a concrete `vX.Y.Z` when it's pinned to a release. |
| **`status`** (release-level) | `planned · active · shipped`. **Distinct from request lifecycle** (`planned | active | review | verified` in PLAN.md): a request can be `verified` (done) while its ledger row is still `planned` (release hasn't shipped). The row flips to `shipped` when the version tags. |

> **Canonical schema home: [ARCHITECTURE.md § Roadmap ledger](../../ARCHITECTURE.md).** That section owns the full column table, tier legend, and the human-adds-rows / gaps-are-normal / planned-runway-only rules. This spec summarizes the ledger as one of the two roadmap layers and adds the `tbd` definition; it does not fork a second authoritative copy.

## Per-version block structure

Each version is its own block. Every block carries:

- **Precision tier** — `full | themed | vision`. Captures the natural gradient (active = `full`, mid-term = `themed`, long-term = `vision`).
- **Outcome slot** *(required for `full` + `themed`, v0.7.2+)* — the goal that sits between Phase and Scope-in. Convergent research (Pichler / Torres / Cagan / Gilad) names this the #1 slot missing from feature-list roadmaps.
- **Phase** — where in the chain the work sits.
- **Scope-in / scope-out** — what's included and explicitly excluded.

## The 9-phase chain

`intent → discover → prototype → spec-refine → mvp → iterate → test → release-prep → release`

Skip notation is allowed (a phase can be marked skipped). Three **meta-phase aliases** — `discover | build | release` — are accepted alongside specific phase values, so a block can start coarse and refine as the work crystallizes. **Prototype** is defined broadly: any artifact that validates a decision against real tooling or against the user (schemas, mocks, wireframes, sample CLI output).

## Tier-aware review gate

An 18-check review gate validates a roadmap block, with checks weighted by tier. Gate guards:

- **No dates in `themed`/`vision` blocks** — Cagan's "#1 sin" of roadmapping.
- **Soft warning at 8+ `full`-tier blocks** — too much false precision.
- **Scope-out push when scope-in ≥ 4** — forces the author to name what's excluded.

## Icebox

Unbound ideas live in the **Icebox** section (renamed from "Bucket list" in v0.7.2 to match the GitHub Projects / Pivotal / Linear idiom). Items are promoted out of the icebox into a versioned block via a 4-step ritual.

## Index mode — shipped-history scaling (v1.23.0+, b18)

Shipped per-version prose blocks accumulate and bloat the file (an agent loading ROADMAP.md to make a planning choice pays for all the past). To bound context cost, ROADMAP.md supports an **index mode** that mirrors the decisions-index pattern:

- **`spectacular roadmap migrate [--dry-run] [--keep N]`** moves shipped prose blocks into per-version files `.spectacular/roadmap/v<X.Y.Z>.md`, keeping the most-recent **N (default 3)** shipped blocks inline. Only blocks whose own `**Status:**` is `shipped` move; planned/active/vision blocks never move.
- ROADMAP.md gains a **`## Shipped`** index — one `- v<X.Y.Z> → roadmap/v<X.Y.Z>.md` line per migrated version. The ledger table stays the build→version spine; the Shipped index is the file-pointer layer for moved prose.
- **Mode detection:** `roadmap/` folder present → index mode; absent → flat mode (all blocks inline).
- **Safety:** per-version files are written before ROADMAP.md is rewritten (no data loss on a partial run); idempotent (re-run moves only newly-ageable blocks); snapshot ROADMAP.md first.
- This finally **enforces** the long-stated "shipped history lives in `CHANGELOG.md`" principle — CHANGELOG holds the facts, `roadmap/v*.md` holds the per-version planning prose, and ROADMAP.md stays lean.

## Doctor integration

`doctor` flags pre-v0.7.1 freeform roadmap shape AND pre-v0.7.2 "Bucket list" naming as **info** (not error) — the roadmap still works, but the newer shape is preferred.

The **`roadmap` doctor area** (v1.23.0+) checks index-mode integrity: orphan `## Shipped` index lines (no matching file), stale per-version files (no index line), and nudges (info) when shipped blocks beyond the keep-window are still inline (prunable via `roadmap migrate`). Flat mode emits only the nudge.
